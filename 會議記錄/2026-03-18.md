## 四、AI 平台選擇

**討論過程：**

Max 提出使用 **Claude Sonnet 4.6（Anthropic）**

Claude 整理選擇原因：

| 考量點 | 說明 |
|---|---|
| 推理能力強 | 財務分析需要理解消費模式、做判斷 |
| 成本效益佳 | 相比 Opus，效能與費用平衡，適合持續運行的預警系統 |
| Tool Use 支援 | 方便 Agent 串接 LineBot API 與資料庫 |
| Context Window 大 | 可保留較長消費歷史在 context 中做分析 |
| 平台一致性 | 與課程 AI 團隊成員（Claude）平台一致 |

**決定：AI 平台選用 Claude Sonnet 4.6 (Anthropic)**

## 五、系統設計細節討論

**資料來源：**

討論手動輸入 vs 銀行/信用卡 API 串接。
考量隱私風險與實作複雜度，決定採用**使用者手動輸入**。

**預警邏輯：**

Max 提出可能加入 Machine Learning，預測未來 7 天消費情況，並搭配 budget 上限。

最終確定**兩層預警機制**：

| 層次 | 機制 | 說明 |
|---|---|---|
| Layer 1 | Budget 上限 | 使用者自設預算，快超過時警告 |
| Layer 2 | ML 預測 | 根據歷史資料預測未來 7 天消費，提前預警 |

**冷啟動規則：**

討論資料不足時的處理方式，決定：
- 至少需要 **7 天**的消費記錄才啟動 ML 預測
- 至少需要 **30 筆**消費資料（每月）才能有效訓練
- 資料不足時：僅使用 Layer 1（Budget 上限）預警

**ML 模型選擇：**

Claude 比較三個選項：
- **Prophet**（Facebook）— 簡單、對時間序列友善、不需要大量資料
- **ARIMA** — 傳統統計，穩定但較難捕捉複雜模式
- **LSTM** — 準確但需要大量資料、訓練成本高

考量每月約 30 筆的資料量，Claude 建議使用 Prophet，Max 同意。

**決定：ML 模型選用 Prophet**

補充說明：組內原本系統已有 CNN + GRU 實作，但考量本專題每月約 30 筆的資料量偏少，改採 Prophet 較為穩健。
